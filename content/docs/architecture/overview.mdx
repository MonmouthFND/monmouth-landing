## High-Level Architecture

```
LLM Runtime → Read State → Simulate → Submit Envelope → Monmouth → Verify Receipt
```

Monmouth separates concerns into three layers:

1. **Layer A (off-chain)**: LLM runtime, planning loop, tool calling, memory, simulation, signing
2. **Layer B (on-chain)**: Deterministic execution, state, capability modules, typed endpoints
3. **Layer C (bridge)**: Policy engine, preflight simulation, attestation hooks, intent receipts

## Why Off-Chain AI?

On-chain ML isn't feasible at consensus speeds. Instead:

- LLMs handle reasoning, planning, and decision-making off-chain
- Monmouth handles deterministic execution, policy enforcement, and settlement
- The chain never runs inference — it verifies outcomes

## Technical Stack

### Commonware Framework

Monmouth is built on [Commonware](https://commonware.xyz), a modular blockchain framework:

- **BFT Simplex Consensus**: Byzantine fault tolerant consensus with 2-second block times
- **Deterministic Execution**: All validators execute the same state transitions in the same order

### Dual-VM Execution

Two virtual machines run side by side in every block:

- **REVM v34**: Full Prague EVM compatibility — deploy any Solidity contract
- **SVM (Solana VM 3.1.9)**: Native Solana program execution — deploy Solana programs natively

Transactions are routed to the correct VM via the [Transaction Envelope](/docs/modules/envelope) module. The `0x4d` magic byte prefix targets SVM; everything else goes to EVM.

### QMDB

Storage is powered by QMDB, a memory-mapped database optimized for blockchain state:

- O(1) state lookups
- Efficient state root computation
- Copy-on-write snapshots for parallel execution

## Twelve Native Modules

Every module is a typed registry with enforced schemas, implemented as Rust crates with `parking_lot::RwLock` for thread-safe concurrent access.

**Core Agent Loop:**
1. [Capability Registry](/docs/modules/capabilities) — tool catalog
2. [Transaction Envelope](/docs/modules/envelope) — explicit tx types + VM routing
3. Simulation / Preview — preflight dry-runs
4. [Policy Engine](/docs/modules/policy) — spending caps, rate limits, allowlists
5. [Intent Receipts](/docs/modules/intent-receipts) — declared intent vs. outcome

**Agent Economy:**
6. [Delegation](/docs/modules/delegation) — scoped session keys
7. State Observation — structured read queries
8. [Memory Anchoring](/docs/modules/memory-anchoring) — on-chain commitments
9. [Coordination](/docs/modules/coordination) — jobs, escrow, settlement
10. Conditional Subscriptions — trigger-based notifications
11. [Attestation](/docs/modules/attestation) — cryptographic verification
12. [SVM Module](/docs/modules/svm) — Solana VM execution

## Agent Transaction Lifecycle

```
1. LLM reads on-chain state via JSON-RPC
2. LLM simulates the action via eth_call (preflight)
3. Agent wraps action in a typed envelope with intent declaration
4. Policy engine validates against spending caps, rate limits, allowlists
5. Transaction routes to EVM or SVM based on envelope type
6. Execution produces state changes + intent receipt
7. Agent reads receipt, compares declared vs. actual outcome
8. Loop repeats
```
